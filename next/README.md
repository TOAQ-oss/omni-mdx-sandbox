# 🧪 Omni MDX: Next.js Sandbox
Welcome to the official sandbox for the Omni MDX project.

This is a lightweight Next.js (App Router) application designed specifically for testing, debugging, and experimenting with the `@toaq-oss/mdx-engine` rendering engine and the Rust core in real time.

This is where we validate the display of interactive scripts for the three podcast series and verify the visual extraction of metadata related to the voice dataset before deploying to production.

## 🎯 Objectives of this environment
1. **Real-Time Validation:** Type Markdown enriched with JSX components and see the Abstract Syntax Tree (AST) generated by Rust instantly transform into a React interface.

2. **Resilience Testing:** Test the `MDXErrorBoundary` system. Insert deliberate errors into component props to verify that the rest of the page continues to display correctly.

3. **Component Prototyping:** Develop and test new custom tags (e.g., podcast-specific audio player components) in isolation from the main application.

## 🚀 Quick Start
First, make sure you have compiled the Rust core from the root of the monorepo (via `make build-web` or `make setup`).

Then, navigate to this folder and start the development server:
```bash
cd sandbox/next
npm run dev
```
Open http://localhost:3000 in your browser. You should see the interactive test interface.

## 📂 Sandbox Architecture
The project follows the standard Next.js structure (`App Router`):

* **`app/page.tsx`**: The main page. It contains the text field and uses `useMemo` to call the WebAssembly parser in an optimized way.

* **`app/components/`**: Contains the “Templates” components injected into the MDX engine.

* **`Speaker.tsx` (Template)**: To simulate and format the dialogues of the voice series.

* **`Note.tsx` (Template)**: For warning and information inserts.

* **`DataChart.tsx` (Template)**: For testing the secure evaluation of complex JavaScript props (arrays, objects).

* **`app/layout.tsx`**: Injects the `MdxConfigProvider` to make components globally available in the test application.

## 🛠️ How to test a new component?
1. Create your React component in `app/components/MyNewComponent.tsx`.

2. Go to the file where your configuration is defined (usually in `app/page.tsx` or a dedicated configuration file).

3. Add it to the provider's `components` dictionary:

```tsx
import { MonNouveauComposant } from './components/MonNouveauComposant';

const mdxConfig = {
  components: {
    // ... autres composants
    MonNouveauComposant
  }
};
```

4. Add the new component `<MyNewComponent myProp="test" />` to your text sent to the MDX engine and see the immediate result!

## 💡 Debugging Tips
* **Browser Console:** If the Rust parser encounters a fatal syntax error (very rare), the `parseError` error will be displayed directly in red on the page interface.

* **React Error Trace:** If you pass invalid data to a component (e.g., `data=“TextInsteadOfAnArray”` in `DataChart`), observe the red box generated by the local `MDXErrorBoundary`, while noting that the rest of the surrounding Markdown content remains intact.
